# 水质运维系统 · 前端优化方案（PC + 小程序双端）

> 版本：v1.0 ｜ 日期：2026-07-22 ｜ 范围：PC 管理台（React19+Vite+Antd5）+ 微信小程序
> 配套交付：`前端优化方案-视觉演示.html`（可交互高保真演示）

---

## 一、现状诊断（基于源码勘察，非臆测）

### 1.1 PC 端

**信息架构**
- 纯顶部导航（无侧边栏、无面包屑、无全局搜索），15+ 个菜单项挤在一排，"运维管理"一个分组下塞了 9 个子项。
- 真正的痛点不在导航本身，而在**功能埋在详情页签里**：试剂质控藏在"站点详情→试剂库存"Tab、备件回收藏在"设备管理→备件库存"Tab、变更审批只在计划状态变化时才显现。新功能做完用户找不到，根因在此。
- 存在孤儿页面：`weekly-plans`（有文件未注册路由）、`workorderPhotos`（空目录）、`reports`（有路由无菜单入口）。
- 顶栏角色徽章只显示"管理员/操作员"二分，但系统实际有 admin/manager/operator/inspector/viewer 五种角色。

**视觉规范**
- 已有双主题令牌体系（`theme/tokens.js`，深色主色 `#00c9a7` / 浅色 `#0d9488`），但**全 src 有 273 处硬编码 hex + 151 处内联 rgba**。
- 令牌采纳严重不均：CockpitPage 用令牌 135 次，而 MaintenancePage（1494 行）仅 6 次、VehiclesPage / EvaluationPage 为 0 次。
- **色板分裂**：`services/thresholds.js` 的阈值色用的是 antd 旧色板 `#52c41a/#faad14/#f5222d`，与令牌体系的 `#10b981/#f59e0b/#ef4444` 完全不同源，导致同一页面两套绿/黄/红。

**性能**
- 构建产物 **3.1MB 单包**（gzip 978KB）。17 个页面全部静态 import，**零 `React.lazy`**，无 `manualChunks`。
- antd + echarts + leaflet + 全部页面代码打进首屏。echarts 实际只有 2 个页面用、leaflet 只有 1 个页面用。
- 3 个死依赖已安装但零引用：`@ant-design/pro-components`、`chart.js`、`react-chartjs-2`。

**工程质量**
- 715 条 lint 告警（多为未使用变量）。
- 6 个超千行巨型页面：cockpit 2207 / alerts 1628 / sites 1511 / maintenance 1494 / equipment 1300 / workorders 1219。
- 共享组件只有 1 个（`AttachmentUpload`），"Table+Modal+Card+STATUS_MAP"的 CRUD 八股在各页重复重写。
- `api.js` 的 401 处理用 `window.location.hash='#/login'`，但应用是 BrowserRouter，**跳转方式不匹配（潜在 bug）**；错误处理仅 console.error，无统一 toast。

### 1.2 小程序端

- 5 个 tabBar（首页/巡检/工单/告警/我的）+ 7 个二级页，结构本身合理。
- 设计令牌只有颜色（17 个 CSS 变量），**缺字号阶梯、间距阶梯、圆角半径变量**，各页面硬编码 8/12/16/24/28rpx。
- `components/` 下 `item-row`、`task-card` 两个组件是**空目录占位**，零自定义组件，全部 Page 内联 wxml；列表行结构在 index/workorder/alert/message 四处复制粘贴。
- 登录守卫（`reLaunch→login`）在 8 个页面 + request 拦截器里重复了 9 遍。
- **品牌色与 PC 不一致**：小程序 `--brand #2B6CFF`（蓝），PC 主色 `#0d9488/#00c9a7`（青绿）。同一个产品两套品牌色。
- 一处配置 bug：`pages/review/` 下文件名是 `review.json`，但注册页面是 `pages/review/view`，应为 `view.json`，导致该页标题配置不生效。
- 正面资产：`services/maps.js` 的"枚举→中文+配色"集中映射层做得好（17 个映射表，界面禁英文），应保留并推广。

### 1.3 双端共性问题（优化的核心抓手）

1. **功能可发现性差**：能力做完埋在页签里，没有统一的"待办聚合"把分散在各模块的待处理事项（待质控/待回收/变更待审/待审核/照片审核）汇总到一处。
2. **设计语言不统一**：PC 青绿、小程序蓝；状态色各自硬编码；无共享的字号/间距阶梯。
3. **组件复用率低**：PC 仅 1 个共享组件，小程序 0 个，大量复制粘贴。

## 二、优化目标与原则

**目标**（按业务价值排序）：
1. 让运维人员和管理者**3 步内触达任何高频功能**，待办事项主动找人而不是人找功能。
2. 建立双端统一、可维护的设计语言，改一处主题全局生效。
3. 首屏加载从 3.1MB 降到 ~600KB 量级，弱网/老旧设备可用。
4. 降低前端维护成本，新增一个 CRUD 页面从"抄 300 行"变成"配 30 行"。

**原则**（与项目一贯理念一致）：
- **业务场景驱动**：不做炫技式改版，每一处改动对应一个真实痛点。
- **不增加一线负担**：优化是让操作更省，不是加流程、加入口。
- **渐进式**：先修 bug 和性能（低风险高回报），再动信息架构，最后沉淀设计系统，避免大爆炸式重构。
- **数据真实**：演示用示意数据会明确标注，不伪造系统状态。

## 三、信息架构与功能可发现性

### 3.1 核心思路：待办聚合，而非单纯改导航

功能"找不到"的根因是**能力埋在详情页签**，而不是菜单层级。因此第一优先级不是重画导航，而是建一个**待办聚合层**，把分散的待处理事项抽出来主动呈现。

**PC：信息中心（工作台）升级为"待办驱动"首页**

在现有 `/`（cockpit）基础上，增设"待办中心"区块，聚合以下事项，每条带**深链直达**操作现场：

| 待办类型 | 来源 | 深链目标 |
|---|---|---|
| 试剂待质控 | `reagent-qc/pending` | 站点详情→试剂库存→自动打开质控弹窗 |
| 备件待回收 | 工单已领料且完工、无回收记录 | 设备管理→备件库存→回收弹窗 |
| 计划变更待审 | `plan_schedules.status=change_submitted` | 计划调度→详情抽屉 |
| 巡检照片待审 | `insp_plan_items.review_status=0` | 照片审核页 |
| 备件/用车申请待审 | 各申请 status=pending | 待办审核页 |

这样"试剂质控""备件回收"这类埋在页签里的功能，**不用改它们的实现位置**，只在首页加一个聚合入口 + 深链参数（如 `?qc=siteId,reagentId` 自动弹窗），即可解决可发现性。这是 ROI 最高的一步。

**PC：补齐导航基础设施**
- 加**面包屑**（如 运维管理 / 设备管理 / 备件库存），解决"我在哪"。
- 加**全局搜索**（`Ctrl/Cmd+K`）：搜站点/工单/设备/页面，直达结果。19 个页面的规模下，搜索比翻菜单快。
- 修正角色徽章为五种角色的完整映射。

**PC：导航结构演进（远期）**
- 19 个页面用顶部导航偏挤。远期建议演进为**侧边导航 + 分组折叠**：监控告警（信息中心/预警/统计分析）、运维作业（工单/巡检计划/计划调度/设备/车辆/试剂）、审核中心（待办审核/照片审核/人员评估）、系统管理（站点/人员/影像档案）。
- 侧边导航放在 P2，因为待办聚合 + 搜索 + 面包屑已能解决 80% 的可发现性问题，不必一上来就动大结构。

**小程序：首页即工作台**
- 现有首页已有"4 格汇总 + 入口卡 + 待检站点 + 待办工单 + 未处理告警"，结构好，保留。
- 增补：把"待我审核"卡升级为**待办聚合卡**（合并审核 + 质控提醒 + 变更通知），按紧急度排序。
- 试剂质控、备件回收目前是 PC 专属。从业务看，更换试剂、换备件都发生在现场，**建议后续把这两个动作做成小程序能力**（本期方案只规划入口位置，不含功能开发）：质控入口放"站点"页试剂区，回收入口放工单完工流程。

### 3.2 深链规范

统一约定 URL 参数协议，让"待办→现场"一步到位：
- PC：`/sites?archive=<siteId>&qc=<reagentId>` → 打开站点详情并自动弹出质控弹窗
- PC：`/equipment?tab=parts&recover=<partId>` → 备件库存页自动弹回收弹窗
- 小程序：`app.globalData` 传参 + 页面 onLoad 消费（现有 selSiteId 模式，规范化命名）

## 四、视觉与 UI 规范化

### 4.1 统一设计令牌体系

**双端共享一套语义令牌**（值可以平台适配，语义必须一致）：

```
品牌色   brand        PC: #0D9488(浅)/#00C9A7(深)   小程序: #0D9488（2026-07-22 已统一，见 §4.2）
成功色   success      #10B981
警告色   warning      #F59E0B
错误色   error        #EF4444
信息色   info         #3B82F6
文本     text-1/2/3/4 四级灰阶
背景     bg-1/2/3/4   四级
边框     border-1/2
圆角     radius-sm/md/lg = 4/8/12 (PC)  8/16/24rpx (小程序)
间距     space-1..6   4/8/12/16/24/32 (PC)  8/16/24/32/48rpx (小程序)
字号     PC: 12/14/16/20/24   小程序: 22/26/30/34/40rpx
```

**状态色统一映射**（双端一致，消灭"两套绿黄红"）：

| 语义 | 场景 | 色值 |
|---|---|---|
| 待处理/进行中 | 待质控、待审批、执行中 | warning 橙 |
| 正常/通过 | 已通过、正常、已完成 | success 绿 |
| 异常/失败 | 不通过、告警、已退回 | error 红 |
| 信息/中性 | 草稿、已归档 | info 蓝 / 灰 |

**落地动作**：
- PC：把 `thresholds.js` 的旧色板改为从 tokens 派生；273 处硬编码 hex 分批替换为令牌（优先重灾区 cockpit/alerts/sites）。
- 小程序：在 app.wxss 补字号/间距/圆角变量，替换硬编码 rpx。

### 4.2 决策记录：品牌色统一为青绿（2026-07-22 已确认并落地）

经确认，双端品牌色统一为青绿族（水质主题契合、PC 为主管理台已采用）。小程序端已完成以下改动，旧品牌蓝 `#2B6CFF` 零残留：
- `app.wxss`：`--brand` → `#0D9488`；`--brand-weak/--brand-line` 同步青绿基色；`--blue` 保留语义蓝并对齐规范 info 色 `#3B82F6`（`.tag-blue/.dot-blue` 不与品牌色混淆）；
- `app.json`：tabBar `selectedColor` → `#0D9488`；
- 5 个 tabBar 选中态图标（`tab-*-on.png`）由蓝重着色为青绿，保留 alpha 抗锯齿；
- `pages/review/view.wxss`、`pages/plan/plan.wxss` 中的品牌蓝硬编码改为引用 `var(--brand)` / `var(--brand-line)`。

### 4.3 组件层建设

**PC：从 1 个共享组件到基础组件库**（`src/components/`）：
- `PageContainer`：统一页头（标题+面包屑+操作区）+ 内容卡片，替代各页自写布局。
- `FilterBar`：统一筛选条（搜索+下拉+日期+重置）。
- `StatusTag`：入参 status + 映射表，统一状态标签渲染（消灭各页手写 STATUS_MAP + 内联色）。
- `FormModal`：统一"新增/编辑"弹窗骨架。
- `DataTable`：封装 Table + 自适应高度（收编现有 `useTableAutoHeight`）+ 空态 + 分页。
- 目标：新增一个 CRUD 页面从抄 300 行降到配置 30~50 行。

**小程序：把两个空目录落地**（`components/`）：
- `item-row`：列表行（标题+副标题+状态标签+右箭头），替换四处复制粘贴。
- `task-card`：待办/任务卡片。
- `sheet`：底部弹层（现有 `.sheet-*` 工具类升级为组件，支持标题/关闭/插槽）。
- 配套：抽一个 `authGuard` 行为（Behavior），替换 9 处重复的登录守卫。

### 4.4 巨型页面拆分

6 个千行页面按"容器 + 子组件"拆分，每个 Tab/弹窗独立成文件。优先拆 maintenance（1494 行，Table×8/Modal×5）与 sites（1511 行）。拆分与组件库建设同步进行，拆出来的重复部分直接沉淀为共享组件。

## 五、性能与加载体验

### 5.1 PC：包体与首屏（P0，立竿见影）

| 措施 | 预期收益 |
|---|---|
| 17 个页面改 `React.lazy` + `Suspense` 路由级分包 | 首屏只加载当前页 |
| `manualChunks` 拆分 vendor：react / antd / echarts / leaflet 各自成块 | 依赖变更不牵连业务代码缓存 |
| echarts、leaflet 改动态 import（仅用到时加载） | 这两个大库不进首屏 |
| 移除死依赖 pro-components / chart.js / react-chartjs-2 | 减小安装与误用风险 |
| 开启 gzip/brotli 压缩插件（vite-plugin-compression） | 传输体积再降 |

目标：首屏 JS 从 3.1MB 降到 **~500–800KB**，首屏时间显著改善。

### 5.2 PC：渲染与感知

- 长列表（工单/告警/附件）引入虚拟滚动（数据量大时）。
- 首屏骨架屏（Skeleton）替代白屏等待。
- 通知轮询 30s 保留，但页面不可见时暂停（`document.visibilitychange`），省电省流量。

### 5.3 小程序

- 当前 12 页 ~2361 行，体积小，**无需分包**。
- 优化点：图片资源压缩、`setData` 只传 diff（避免整列表重传）、列表分页加载（message 页已有上拉加载，推广到工单/告警列表）。

## 八、风险与取舍

- **侧边导航是可见大改**：放 P2，先用待办聚合+搜索+面包屑解决主要矛盾，避免为了"看起来专业"而增加一线学习成本。
- **品牌色统一已确认**（2026-07-22）：统一为青绿族，小程序端令牌/tabBar/选中态图标已同步，零残留。
- **令牌替换有回归风险**：分批替换 + 每批后回归对应页面，不做一次性全局替换。
- **组件库忌过度设计**：只抽"已重复 3 次以上"的模式，不预设通用框架，避免自嗨。
- **性能数据为估算**：~600KB 为目标值，实际以构建报告为准，P0 完成后复测。
