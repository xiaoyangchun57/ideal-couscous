# 水质运维系统优化方案：从"功能堆砌"到"场景驱动"

> 日期：2026-07-21
> 前提：不大幅重构后端与数据模型，在现有 React+Antd / 微信小程序 / Flask 技术栈上做渐进式调整

---

## 一、问题诊断：为什么被评价为"自我感动"和"反向增负"

### 1.1 根因

系统按**数据实体**组织功能（站点、设备、试剂、车辆、工单、告警、影像、审核……），每个实体一个独立页面，各自有增删改查。这是开发者视角，不是使用者视角。

真实运维人员的一天是这样的：

```
早上看今天去哪几个站 → 开车到站 → 打卡 → 按检查项逐项看/拍/记 → 
发现异常拍照上报 → 回站路上处理工单 → 晚上确认数据是否正常
```

他不需要"打开设备管理页面 → 搜索站点 → 找到仪器 → 查看历史"这种四步操作。他需要的是：**到了站点，所有跟这个站相关的待办、数据、设备状态一屏可见**。

### 1.2 "反向增负"的具体表现

| 现象 | 本质 |
|------|------|
| 巡检照片不足不能提交，但上传入口不明显 | 门禁逻辑有了，但操作路径没铺好——拦住了人却没指路 |
| 试剂管理独立成页，与站点/仪器脱节 | 按实体建模，但用户心智是"这个站的这台仪器该换试剂了" |
| 驾驶舱数据健康度按站点在线率算 | 指标定义脱离考核实际（应为正常参数量/总参数量） |
| 审核中枢4个tab，管理者要逐个点开 | 缺少"今天最需要我处理的3件事"的聚合视图 |
| 周计划存在但流程不完整 | 缺少"物资确认→管理员审核→任务下发"的完整闭环，排了计划但车/备件没落实 |
| 非负责人看到不相关的推送和任务 | 权限隔离未做，信息噪音大 |

### 1.3 一个判断标准

对任何功能问一个问题：**如果这个功能明天消失，用户会不会主动来问？**

- 会问：周计划排程、巡检执行、工单接单、告警通知、照片上传、车辆/备件确认 → 核心
- 不会问但需要存在：审核流程、备件库存管理、操作日志、影像归档 → 支撑
- 消失了反而轻松：独立的试剂主数据页面（应收编入站点）、影像档案独立管理页（应改为搜索视图） → 应收编或降级

## 二、角色场景适配：两类角色的真实工作流重建

### 2.1 一线运维人员（5-10人，每人固定负责3-5个站）

**核心心智模型：计划先行，站点为中心，任务为驱动**

```
┌─────────────────────────────────────────────────────┐
│  一周的工作流                                         │
├─────────────────────────────────────────────────────┤
│  ⓪ 排程（周初/周末）：                                │
│     排下周计划（去哪几个站、干什么）                    │
│     → 确认用车需求（哪天用、是否与他人冲突）            │
│     → 确认备件需求（这个工单要带什么备件）              │
│     → 提交管理员审核                                  │
│     → 管理员判断合理性 + 物资调配 → 通过              │
│                                                     │
│  ① 早上：看今日任务（哪几个站、每站干什么）            │
│  ② 到站：GPS打卡 → 拍照权限解锁                      │
│  ③ 执行：逐项检查 → 正常打勾 / 异常拍照+描述         │
│  ④ 顺路：这个站有未关工单？一并处理（降低运维成本）    │
│  ⑤ 附带：试剂快到了？顺手登记。设备有问题？发起工单    │
│  ⑥ 离站：提交巡检（照片自动校验够了没）               │
│  ⑦ 路上/晚上：处理工单、确认告警、看消息              │
└─────────────────────────────────────────────────────┘
```

**关键业务逻辑：巡检迎合工单。** 巡检是定期无目的计划，工单是针对特定问题。为降低运维成本，排巡检计划时系统应主动提示"该站有未关闭工单，建议顺路处理"，让一趟出行解决多个问题。

**高频操作（每天/每周多次，主要在移动端完成）：**
- 排周计划 + 确认用车/备件需求 + 提交审核
- 查看今日/本周任务列表
- 到站打卡（GPS自动）→ 拍照权限解锁
- 逐项检查提交（正常/异常+拍照）
- 查看/确认告警
- 工单接单→到场→拍照→提交

**低频但必须的操作：**
- 试剂更换登记
- 备件申请
- 用车申请
- 校准记录

**设计原则：**
- 打开即见"今天要干什么"，不需要选菜单
- 所有操作以站点为入口聚合，不散落在不同页面
- 能点选不手输，能拍照不打字
- 门禁（如照片不足不能提交）要**提前告知**而非提交时才报错

### 2.2 管理人员（1-2人）

**核心心智模型：异常驱动，决策支持**

```
┌─────────────────────────────────────────────────────┐
│  管理者的关注层次                                     │
├─────────────────────────────────────────────────────┤
│  第一层（每天扫一眼）：                               │
│    - 今天谁去了哪、完成了没                            │
│    - 有没有未处理的告警/异常                           │
│    - 数据健康度（按人、按站）                          │
│                                                     │
│  第二层（需要时深入）：                               │
│    - 某个站最近怎么了（趋势、历史异常）                │
│    - 某个人的工作量和完成质量                          │
│    - 审核待办（巡检照片、工单关单、备件申请）          │
│                                                     │
│  第三层（周期性）：                                   │
│    - 报表导出、考核数据                               │
│    - 排班/计划调整                                   │
│    - 迎检资料归档                                    │
└─────────────────────────────────────────────────────┘
```

**高频操作：**
- 看概览（谁在干什么、哪里有问题）
- 审核（通过/退回，附带意见）
- 处理告警升级

**低频操作：**
- 排班编排
- 规则/阈值配置
- 报表生成
- 用户/权限管理

**设计原则：**
- 首页是"决策仪表盘"而非"功能入口列表"
- 审核操作尽量一步完成（列表内直接通过/退回，不必进详情）
- 数据健康度按考核口径（正常参数量/总参数量），按人分组展示
- 异常置顶，正常折叠

## 三、功能精简优化：PC端模块重组方案

### 3.1 现有17个页面的处置建议

| 现有页面 | 处置 | 理由 |
|----------|------|------|
| 驾驶舱 CockpitPage | **保留，重构内容** | 改为真正的决策视图：今日执行概览+异常聚合+数据健康度（按人） |
| 站点 SitesPage | **保留，改为聚合入口** | 站点详情页整合：实时数据+设备+试剂+近期巡检+工单，一屏可见 |
| 告警 AlertsPage | **保留** | 核心功能，优化列表信息密度和筛选 |
| 工单 WorkOrdersPage | **保留** | 核心功能 |
| 巡检 V2 inspection | **保留** | 核心功能 |
| 审核中枢 AuditPage | **保留，简化交互** | 列表内快捷操作，减少"进详情→看→返回"的来回 |
| 周计划 WeeklyPlansPage | **保留并强化** | 高频核心功能，补全"物资确认→管理员审核→任务下发"闭环 |
| 设备 EquipmentPage | **收编入站点详情** | 设备天然属于站点，独立页面使用频率极低 |
| 试剂 ReagentMasterPage | **收编入站点详情** | 试剂属于站点仪器，在站点内管理 |
| 维保 MaintenancePage | **降级为工单子类型** | 维保本质是一种计划性工单，不必独立模块 |
| 车辆 VehiclesPage | **保留，融入计划流程** | 用车确认是计划排程的必要环节，需冲突检测和主管审批 |
| 备品备件（新增独立视图） | **公司级库存管理** | 区别于站点设备，备件是公司资产，需入库/出库/回收/盘点 |
| 影像档案 PhotosPage/ArchivePage | **改为审核+搜索视图** | 照片需审核（是否现场拍摄），归档按站点/时间/来源自动分类 |
| 分析 AnalysisPage | **保留但降优先级** | 管理者偶尔用，不占核心位置 |
| 报表 ReportsPage | **保留但降优先级** | 周期性使用 |
| 用户 UsersPage | **保留，仅管理员可见** | 权限隔离后普通用户看不到 |
| 登录 LoginPage | 保留 | — |
| 评价 EvaluationPage | **视情况保留** | 如果是考核迎检用，保留但降优先级 |

### 3.2 重组后的PC端菜单结构（主要服务管理者）

> 注意：PC端运维人员使用频率低，主要服务管理者的审核、监控、决策场景。运维人员的高频操作在移动端完成。

```
├── 工作台（原驾驶舱，重构）
│     今日执行概览 / 异常告警聚合 / 数据健康度（按人） / 待我审核
├── 计划与调度（核心，高频）
│     ├── 周计划审核（运维人员提交→管理员审批，含用车/备件确认）
│     ├── 车辆调度（用车申请、冲突检测、审批）
│     └── 态势看板（执行进度、计划覆盖率）
├── 巡检管理
│     ├── 执行记录
│     └── 巡检模板/检查项配置
├── 工单中心（合并工单+维保）
│     ├── 全部工单（类型筛选：故障/维保/应急）
│     └── 工单跟踪（状态流转、关联巡检）
├── 站点管理（聚合入口）
│     ├── 站点列表 → 站点详情（数据/设备/试剂/影像/历史）
│     └── 校准记录
├── 物资管理
│     ├── 备品备件库存（公司级：入库/出库/回收/盘点）
│     └── 备件使用记录（关联到具体工单/站点）
├── 告警中心
│     ├── 告警事件
│     ├── 数据审核（L1/L2/L3）
│     └── 规则配置
├── 审核中枢（简化交互）
│     巡检照片审核 / 工单关单审核 / 备件申请审核
├── 影像审核与查询
│     ├── 待审核影像（是否现场拍摄、是否合规）
│     └── 影像搜索（按站点/时间/来源/操作人）
├── 报表与分析
│     ├── 报表导出 / 考核数据
│     └── 数据分析
└── 系统设置（仅管理员）
      ├── 用户与权限
      └── 系统配置
```

**一级菜单从17个调整为10个。** 核心变化：周计划升级为"计划与调度"一级模块（含车辆）；备品备件独立为公司级物资管理；影像从"档案管理"变为"审核+搜索"。

### 3.3 一线人员的使用场景分配

> 核心原则：**移动端是一线人员的主战场，PC端仅偶尔查看。**

**移动端（高频，每天使用）：**
- 排周计划 + 确认用车/备件 + 提交审核
- 今日任务 → 到站打卡 → 巡检执行 → 拍照 → 提交
- 工单接单/处理/关单
- 告警确认
- 消息通知

**PC端（低频，偶尔使用）：**
- 查看自己的历史记录和统计
- 补录/编辑在手机上不方便操作的内容
- 不需要看到：用户管理、规则配置、审核中枢、全局驾驶舱

## 四、界面交互改进：不重构前提下的快速提升路径

### 4.1 已有的好基础

你已经做了：
- 前端UI统一优化样式规范（MetricCard + AuditToolbar + 三层结构）
- 移动端完整Design System（色彩/字体/间距/圆角/组件规范）
- check_ui_zh.py 中文合规闸门
- useTableAutoHeight 自适应表格

这些是好的。问题不在于"没有规范"，而在于**规范覆盖的是组件层，缺少页面层的信息架构指导**。

### 4.2 PC端快速提升（按投入产出比排序）

**第一优先级：工作台重构（1-2天）**

当前驾驶舱的问题：面板多但信息散，看完一圈还是不知道"今天最该关注什么"。

改为：
```
┌────────────────────────────────────────────────────────┐
│  顶部：今日一句话摘要                                    │
│  "5人出勤，已完成12/18站巡检，2条未处理告警，1单待审核"   │
├────────────────────────────────────────────────────────┤
│  左区（60%）：                                          │
│  ┌──────────────────────────────────────────┐          │
│  │ 待我处理（审核/告警/异常）                  │          │
│  │ 按紧急程度排序，最多显示5条，可展开          │          │
│  │ 每条可一键操作（通过/退回/确认/转工单）      │          │
│  └──────────────────────────────────────────┘          │
│  ┌──────────────────────────────────────────┐          │
│  │ 今日执行进度                              │          │
│  │ 按人分组：张三 3/4站 ✓ | 李四 2/3站 ⚠    │          │
│  └──────────────────────────────────────────┘          │
├────────────────────────────────────────────────────────┤
│  右区（40%）：                                          │
│  ┌──────────────────────────────────────────┐          │
│  │ 数据健康度（按人，考核口径）                │          │
│  │ 张三 96.7% | 李四 98.2% | 王五 91.5% ⚠   │          │
│  └──────────────────────────────────────────┘          │
│  ┌──────────────────────────────────────────┐          │
│  │ 近7天告警趋势（折线图，按等级分色）         │          │
│  └──────────────────────────────────────────┘          │
└────────────────────────────────────────────────────────┘
```

**第二优先级：站点详情页聚合（1-2天）**

当前站点详情是分散的（设备一个页、试剂一个页、影像一个页）。改为站点详情内用Tab聚合：

```
站点详情
├── 概览（实时数据 + 最近异常 + 设备状态摘要）
├── 设备与试剂（设备列表，展开可见试剂状态和更换记录）
├── 巡检记录（该站近期巡检，含照片缩略图）
├── 工单历史
├── 影像资料（该站所有照片，按时间/来源筛选）
└── 校准记录
```

这样运维人员或管理者点进一个站，所有信息一屏可达，不需要在5个页面间跳转。

**第三优先级：审核交互简化（0.5-1天）**

当前审核要"点进详情→看→返回→下一个"。改为：
- 列表每行增加快捷操作按钮（通过✓ / 退回✗）
- 退回时弹出小气泡输入原因，不必打开Modal
- 照片在列表行内可展开缩略图预览，不必进详情
- 批量操作：勾选多条 → 一键通过

**第四优先级：全局视觉统一收尾（已有规范，执行即可）**

你已有的样式规范（三层结构、MetricCard、AuditToolbar、tokens色彩）是好的，按执行清单逐页落地即可。补充几点：

- 所有状态标签统一为胶囊Tag（你移动端规范里已定义，PC端同步）
- 列表列宽统一调整（你需求里提到的）
- 滚动条样式统一（参考驾驶舱站点面板）
- 空状态统一（无数据时显示引导而非空白）

### 4.3 移动端快速提升

你的移动端UI设计升级方案已经非常完整（Design Tokens、组件规范、页面布局），执行层面建议：

**关键改进点（按用户痛感排序）：**

1. **首页即任务**：打开小程序第一眼是"今天去这几个站"，不是统计数字。统计数字缩小为顶部一行，主体是任务卡片列表。

2. **巡检执行的拍照体验（打卡后才能拍照是反作弊设计，保留）：**
   - 到站后GPS自动触发打卡 → 打卡成功 → 相机按钮从灰色解锁为可用 + 轻提示"已打卡，可以拍照"
   - 检查项标记异常时，自动弹出相机（不是让用户找上传按钮）
   - 照片缩略图实时显示在检查项下方
   - 提交前明确提示"还需X张照片"（提前告知，不是提交时才拦）
   - 管理者审核时可看到每张照片的打卡时间戳和GPS，验证现场拍摄

3. **工单详情的操作路径**：
   - 底部固定一个主按钮，文案随状态变化（接单→到场→开始处置→提交）
   - 拍照按钮做大做明显（88rpx以上，带相机图标）
   - 减少信息展示，详情页只放"这个工单要干什么+当前状态+操作按钮"

4. **弱网体验**：
   - 提交失败时不要白屏或转圈，显示"已保存本地，网络恢复后自动上传"
   - 照片上传显示进度条，失败显示重试按钮

## 五、不重构前提下的实施路径

### 5.1 分阶段推进（每阶段可独立交付价值）

| 阶段 | 内容 | 工期 | 用户感知 |
|------|------|------|----------|
| **Week 1** | 权限隔离（按人过滤站点/任务/推送）+ 移动端首页重构（今日任务为核心） | 3-4天 | "终于只看到跟我有关的了" |
| **Week 2** | 周计划闭环（排程→用车/备件确认→管理员审核→任务下发）+ 巡检迎合工单提示 | 4-5天 | "排计划、要车、要备件一条龙了" |
| **Week 3** | 移动端巡检/工单体验打磨（打卡→拍照解锁→提交引导）+ 影像审核流程 | 3-4天 | "手机上干活顺多了" |
| **Week 4** | PC端管理者工作台重构 + 站点详情聚合 + 物资管理独立视图 | 4-5天 | "审核和监控一目了然" |

### 5.2 不动什么（保护已有投入）

## 六、关于"场景脱节"的根本性建议（已根据业务纠正修订）

技术方案能解决"怎么用"的问题，但"该不该有"和"怎么串"的问题需要回到业务本身。几点建议：

**1. 周计划是整个系统的"发令枪"，必须做好。** 它是运维人员主动排程、管理员审核调配的核心流程。做好的标准是：排计划时自动显示"这个站有未关工单，建议顺路"、"这天车辆已被张三预约"、"这个备件库存只剩1个"——让排计划的过程本身就是资源协调的过程，而不是排完了再单独去申请车和备件。

**2. "巡检迎合工单"是降低运维成本的核心智能点。** 巡检是定期无目的计划，工单是特定问题。系统应该在排巡检计划时主动匹配：哪些站有未关闭工单？哪些工单的备件正好可以这次带过去？哪些告警需要现场复核？让一趟出行解决多个问题，这才是"智能化"的真正价值。

**3. 打卡后才能拍照是正确的反作弊设计，体验上让它"自然"而非"被拦"。** 到站→GPS自动打卡→相机解锁，整个流程应该是无感的。管理者审核照片时能看到时间戳和GPS，这是信任基础。不要为了"体验顺滑"而弱化这个机制。

**4. 备品备件要区分"站点消耗"和"公司资产"两个视角。** 站点详情里看的是"这个站的仪器用了什么备件、什么时候换的"；物资管理里看的是"公司还有多少库存、谁领了什么、旧件回收了没"。两者通过出库/使用记录关联，但各自有独立的管理需求。

**5. 数据健康度改为考核口径后，要让人能"drill down"。** 管理者看到某人91.5%，点进去要能看到：哪个站、哪个参数、哪条数据拉低了分数。否则数字只是数字，不产生行动。

**6. 影像审核的重点是"是否现场拍摄"。** 管理者对照片真实性非常重视。审核界面应突出显示：拍摄时间（与打卡时间的差值）、GPS位置（与站点坐标的距离）、来源（哪次巡检/哪个工单）。异常项（如打卡后30分钟才拍、GPS偏离站点500米）应标红提醒。

**7. 找1-2个一线运维人员做半天影子观察。** 不是问他们"你需要什么功能"（他们说不出来），而是看他们实际怎么干活：几点出门、到了站点先干什么、手机打开什么app、跟主管怎么沟通、遇到异常怎么处理。半天观察胜过十次需求访谈。

## 七、总结

你的系统技术基础扎实（后端闭环、数据模型完整、双端架构清晰），问题出在**功能组织逻辑**和**信息架构**层面。核心调整是：

1. **以周计划为起点串联全链路**：排程→资源确认（车/备件）→审核→任务下发→执行→上报→审核，一条线贯穿
2. **巡检迎合工单，降低运维成本**：排计划时智能匹配未关工单、待复核告警、待换备件，让一趟出行价值最大化
3. **移动端是一线人员的主战场**：所有高频操作在手机上完成，PC端服务管理者的审核和决策
4. **从"实体管理"转向"场景驱动"**：站点是执行锚点，计划是调度锚点，两者串联
5. **打卡→拍照是信任链**：反作弊设计保留，体验上做到无感自然
6. **备品备件双视角**：站点看消耗，公司看库存，通过出库记录关联
7. **影像审核重在真实性**：时间戳+GPS+来源三重验证，异常标红

这些调整不需要重写后端，不需要换技术栈，主要是前端流程串联、菜单调整、权限过滤和交互优化。4周内可以完成主体改造，用户感知会有质的变化。
