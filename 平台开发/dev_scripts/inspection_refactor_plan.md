# 巡检计划模块改造方案

## 一、当前问题诊断

### 问题1：角色链路断裂
当前模块不知道"谁在做什么阶段"——
- 方案模板谁设的？管理员
- 计划谁生成的？系统
- 周计划谁创建的？运维人员
- 谁审核？不知道

**结果**：每个页面看起来独立，但串联起来缺角色切换的概念。

### 问题2：数据模型冲入
两套并行系统（`inspection_templates` + `insp_plans` vs `inspection_v2_*`）的API路径用了相同的 `/api/inspection-v2/` 前缀，但实际操作不同的表。开发者都分不清，用户更不懂。

### 问题3：概念不统一
- 频次：模板用 `high/mid/low`，计划用 `weekly/monthly`，是两个体系
- 分类：模板检查项有 `站房环境/设备运维`，但模板本身分类有 `水文站/蒸发站`，与水质运维场景不符
- 周计划独立存在 + 独立的审批流程，和巡检计划功能完全重叠

## 二、改造方案（4阶段工作流）

```
┌─────────────────────────────────────────────────┐
│  阶段一：准备（运维人员）                        │
│  创建/查看计划 → 关联车辆 → 预申报备件         │
│  → 查看站点历史异常 → 出发                       │
├─────────────────────────────────────────────────┤
│  阶段二：到达站点（运维人员）                    │
│  GPS签到 → 逐项执行检查 → 填结果+拍照          │
│  发现异常→一键上报告警+工单                     │
├─────────────────────────────────────────────────┤
│  阶段三：离场（运维人员）                        │
│  GPS签退 → 提交结果 → 备件消耗确认             │
├─────────────────────────────────────────────────┤
│  阶段四：审核（主管）                            │
│  审核结果 → 确认备件消耗 → 计划办结归档        │
└─────────────────────────────────────────────────┘
```

### 数据绑定
- **计划 ↔ 车辆**：创建/编辑时选择，列表显示车牌
- **计划 ↔ 站点**：一个计划覆盖多个站点，各站点独立检查项
- **计划 ↔ 备件**：创建时推导建议备件清单，可预申报审批
- **检查项 ↔ 照片**：执行时拍照上传，审核时查看
- **检查项 ↔ 告警**：异常上报自动创建告警+工单

## 三、具体改造项

### 3.1 数据层清理（已完成）
- ✅ 删掉 `inspection_v2_*` 幽灵表（6张）
- ✅ `insp_plans` 加 `vehicle_id` 列
- ✅ 频次统一：`high→daily`, `mid→weekly`, `low→monthly`
- ✅ 分类清理：`环境→站房环境`
- ✅ `inspection_configs` 99条重复→2条

### 3.2 方案模板Tab
**当前**：
- 模板分类有 `水文站/蒸发站` 等与水质无关的分类
- 检查项频次用 `high/mid/low` 
- 新建模板后前端刷新可能不及时

**改后**：
- `inspection_templates.category` → 只保留 `水质监测/站院环境`
- `inspection_template_items.category` → 保留 `站房环境/设备运维/质控校准/台账登记`
- `inspection_template_items.frequency_level` → 统一为 `daily/weekly/monthly`

### 3.3 巡检计划Tab（替代周计划）
**当前**：巡检计划Tab + 周计划Tab 分离且重叠

**角色修正**：计划由运维人员创建/执行，主管仅做审核。

**改后**：一个PlanTab搞定所有

| 功能 | 当前 | 改后 |
|---|---|---|
| 创建者 | 系统自动 | 运维人员手动创建，系统自动生成作为补充 |
| 车辆选择 | 周计划有 | PlanTab直接选 |
| 手动创建 | 无 | "手动创建"弹窗（选名称/负责人/周期/车辆/站点） |
| 状态流转 | 周计划有 | PlanTab直接支持 draft→submitted→approved→rejected |
| 站点历史概览 | 无 | 计划详情顶部显示该站点近30天告警 |
| 备件预申报 | 无 | 创建时根据检查项推导备件清单，可预申报 |
| 编辑站点 | PlanTab已有 | 保留 |
| 关联车辆显示 | 无 | 列表+详情都显示车牌号 |

### 3.4 巡检配置Tab
**当前**：99条无区别的配置，用户不知道意义

**改后**：
- 保留配置概念但缩减为 `site_type→template_id` 的简洁映射
- 顶部加说明："配置用于定义站点类型与模板的匹配关系，一键生成时自动应用"
- 种子数据覆盖6种站点类型各1条

### 3.5 统一术语表

| 模块 | 当前不一致 | 统一后 |
|---|---|---|
| 频次 | high/mid/low + weekly/monthly | daily / weekly / monthly |
| 分类 | 水文站/蒸发站+站房环境/设备运维 | 站房环境/设备运维/质控校准/台账登记 |
| 计划状态 | draft/active + draft/submitted | draft→active→completed→archived |
| 车辆 | 周计划有 | 所有计划 |

## 四、新增功能详细描述

### F1. 巡检中异常上报（P0）
**场景**：现场检查时发现异常 → 点一下"上报" → 自动生成告警+工单。

复用现有 `/api/manual-reports` 后端逻辑，增加 `plan_item_id` 关联。执行时一键创建告警+工单，不走额外流程。

### F2. 备品备件预申报（P0）
**场景**：出发前预判需要备件 → 申请 → 审批 → 库存预留 → 现场领用。

创建计划时或计划详情中根据检查项推导建议备件清单，一键提交 `parts_requests`，审批通过后库存锁定。

### F3. 站点历史异常概览（P1）
**场景**：出发前看站点"病历" → 近30天告警 + 上次异常项。

计划详情页顶部多一个"站点近况"区域，只用 JOIN 查询 `alerts` 表最新5条，低代码高收益。

### F4. 签到/离场打卡（P1）
**场景**：到达签到 → 开始执行 → 完成签退。

`insp_plan_items` 加 `check_in_time/check_out_time/check_in_gps`，前端获取浏览器 Geolocation API，纯前端功能无后端改动。

### F5. 检查项关联备件SKU（P1）
**场景**：检查项执行时判定"需更换" → 一键扣库存。

`inspection_template_items` 加 `part_sku` 字段，执行时标记"需更换"自动扣 `parts_inventory`。

### F6. 质控品携带提醒（P2）
**场景**：出发前列出需带的试剂/标液清单。

根据检查项类型推导所需试剂，从 `reagents` 表查出规格，生成 checklist。

## 五、实施优先级

| 批次 | 内容 | 预估工时 |
|---|---|---|
| 第一批 P0 | 异常上报 + 备件预申报 | 2天 |
| 第二批 P1 | 历史概览 + 签到打卡 + 备件SKU | 2天 |
| 第三批 P2 | 质控品提醒 | 1天 |

## 六、工作流说明（给用户看）

```
你（运维人员）：
1. 打开"巡检计划"→ 新建一个周计划
2. 选这周要去的站点 + 关联车辆 + 查看备件清单
3. 提交给主管审批 → 主管通过

你（运维人员）出发前：
4. 打开计划详情 → 看"站点近况"了解最近出的问题
5. 看"建议携带清单"→ 去仓库领备件和质控品

你（运维人员）到站：
6. GPS签到 → 逐项检查
7. 每个检查项填结果、拍照
8. 发现异常？点"上报异常"→ 自动通知主管+生成工单

你（运维人员）离场：
9. GPS签退 → 确认备件使用情况 → 提交

主管：
10. 打开"待办审核"→ 看到提交的巡检结果
11. 逐项审：通过就完事，驳回退回重做
12. 确认备件消耗 → 计划办结归档
```
