# 巡检计划体系设计（统一版）

> 日期：2026-07-22
> 版本：v2.0（替代原《周计划模块细化设计》）
> 核心变更：周计划不再作为独立模块，而是并入统一的巡检计划分层体系（周/月/季/年）
> 前提：不重写后端，在现有 `insp_plans`/`insp_plan_items` 基础上增量扩展

> **实现校准备忘（P1 落地，2026-07-22）：**
> 1. **调度层表名定为 `plan_schedules`**（原设计的 `inspection_schedules` 已被现有"检查项到期排程"表占用，避免冲突）。
> 2. **检查项模板匹配复用现有 `inspection_templates.frequency` 列**（weekly/monthly/…，与 schedule_type 天然对齐），不再新增 schedule_type 列。
> 3. **审批通过生成的 `insp_plans` 直接为 `active`**（计划已审批，不做二次审批），并通过新增列 `insp_plans.plan_schedule_id` 溯源到调度计划（变更重建时按此列定位）。

## 一、设计理念

### 1.1 三个根本性认知（来自业务纠正）

**① 周计划就是最高频的巡检任务，不是独立模块。**
每位运维人员每周需对所有负责站点巡检一次（周频次固定）。此外还有月度、季度、年度巡检，巡检重点不同。它们本质是同一个"巡检计划"按频次分层，周计划只是最频繁的一档。

**② 用车/备件是出发前的事，不是处置后的事。**
人员驻地统一，但站点分布广泛。接单/排程后、出发前必须落实"怎么去、带什么"。资源协调是排程的有机组成部分。

**③ 审批需要决策支撑，不是简单点通过。**
计划由运维自己排，但管理者审批时要能判断合理性：站点近期情况如何？严重问题的站是否被排在后面？路线有没有折返浪费？

### 1.2 体系定位

```
巡检计划体系 = 调度层（排程+资源协调+审核） + 执行层（检查项+打卡+拍照+提交）
```

调度层服务于所有频次（周/月/季/年），执行层复用现有 `insp_plans`/`insp_plan_items`。

## 二、巡检计划分层模型

### 2.1 四个频次层级

| 层级 | 频次 | 覆盖范围 | 巡检重点 | 检查项特征 |
|------|------|---------|---------|-----------|
| **周巡检** | 每周 | 全覆盖（所有负责站点） | 常规检查、读数、外观、清洁、基础维护 | 项多但每项简单，拍照多 |
| **月巡检** | 每月 | 部分/全部站点 | 校准、质控、易损件检查 | 含质控验证项 |
| **季巡检** | 每季 | 重点站点 | 深度维护、性能验证 | 项少但深入 |
| **年巡检** | 每年 | 全部站点 | 全面检修、设备评估、备件大修 | 备件需求大、耗时长 |

### 2.2 统一数据模型

**核心思想：用一张 `inspection_schedules`（调度层）+ 现有 `insp_plans`（执行层）表达所有频次。**

```sql
-- 调度层：巡检计划排程（统一所有频次）
CREATE TABLE IF NOT EXISTS inspection_schedules (
    id INTEGER PRIMARY KEY AUTOINCREMENT,
    user_id INTEGER NOT NULL,              -- 排程人（运维）
    schedule_type TEXT NOT NULL,           -- weekly/monthly/quarterly/yearly
    period_start DATE NOT NULL,            -- 周期开始（周一/月初/季初/年初）
    period_end DATE NOT NULL,              -- 周期结束
    plan_data TEXT NOT NULL DEFAULT '{}',  -- JSON: 日期→站点列表（见2.3）
    -- 资源协调
    vehicle_days TEXT DEFAULT '{}',        -- JSON: 日期→vehicle_id
    spare_parts TEXT DEFAULT '[]',         -- JSON: 备件需求列表
    work_order_ids TEXT DEFAULT '[]',      -- JSON: 顺路处理的工单
    -- 审核
    status TEXT DEFAULT 'draft',           -- draft/submitted/approved/rejected/modifying/change_submitted/archived
    approver_id INTEGER,
    submitted_at TIMESTAMP,
    approved_at TIMESTAMP,
    reject_reason TEXT,
    -- 变更
    version INTEGER DEFAULT 1,
    change_reason TEXT,
    previous_plan_data TEXT,
    -- 其他
    remarks TEXT,
    tasks_generated INTEGER DEFAULT 0,     -- 是否已生成执行任务
    created_at TIMESTAMP DEFAULT (datetime('now','localtime')),
    FOREIGN KEY (user_id) REFERENCES users(id)
);
```

### 2.3 plan_data 结构（按频次自适应时间粒度）

**周巡检**（按天排）：
```json
{
  "2026-07-27": {"sites": [3, 7], "notes": "麻山顺路工单#42"},
  "2026-07-28": {"sites": [12], "notes": ""},
  "2026-07-29": {"sites": [], "notes": "休息"}
}
```

**月/季/年巡检**（按具体日期排，粒度相同，只是周期更长）：
```json
{
  "2026-08-05": {"sites": [3], "notes": "麻山月度校准"},
  "2026-08-18": {"sites": [7, 12], "notes": "质控"}
}
```

> 统一用"日期→站点"结构，周/月/季/年只是 period_start/period_end 跨度不同，前端按 schedule_type 切换日历视图（周视图/月视图）。

### 2.4 检查项模板按频次区分

```sql
-- 扩展现有 insp_templates，增加频次维度
ALTER TABLE insp_templates ADD COLUMN schedule_type TEXT DEFAULT 'weekly';
-- 同一站点可有不同频次的模板：
--   麻山站-weekly模板：常规检查项（外观/读数/清洁）
--   麻山站-monthly模板：校准+质控项
--   麻山站-yearly模板：全面检修项
```

生成执行任务时，按 `schedule_type` 匹配对应模板的检查项集合。

### 2.5 全覆盖校验（按频次规则）

```python
def validate_coverage(schedule):
    user_sites = get_user_sites(schedule.user_id)  # 该运维负责的所有站
    planned_sites = set()
    for day_data in schedule.plan_data.values():
        planned_sites.update(day_data["sites"])

    if schedule.schedule_type == 'weekly':
        # 周巡检要求全覆盖
        missing = set(user_sites) - planned_sites
        if missing:
            return {"ok": False, "warning": f"本周未覆盖：{missing}，周巡检要求全覆盖"}
    # 月/季/年按各自覆盖规则（可配置）
    return {"ok": True}
```

## 三、调度层：排程 → 资源协调 → 审核

### 3.1 运维人员排程流程（移动端为主）

```
选择周期类型（周/月/季/年）和周期范围
    │
    ▼
逐日安排站点（日历视图，周视图/月视图）
    │  ⚡ 实时智能建议（见3.2）
    │  ⚡ 全覆盖校验（周巡检漏站提示）
    │  ⚡ 路线折返提示（见五.3）
    ▼
出发前资源协调
    ├── 🚗 用车：选车 + 冲突检测（不同天可选不同车）
    └── 🔧 备件：智能建议 + 库存校验 + 手动增减
    │
    ▼
填写备注 → 提交审核
    │
    ▼
管理员审核（PC端，见第四节）
    │
    ├── 通过 → 自动流转（生成任务+锁车+备件出库+通知）
    └── 退回 → 填原因 → 运维修改重交
```

### 3.2 智能建议（排程时实时提示，一套数据双端复用）

| 建议类型 | 触发 | 提示内容 |
|---------|------|---------|
| 工单顺路 | 选站时查该站未关工单 | "麻山有1条紧急工单（氨氮超标，已9天），建议顺路处理并排在前面" |
| 告警复核 | 选站时查该站待复核告警 | "山口岩有黄色告警待现场复核" |
| 试剂临期 | 选站时查该站试剂余量 | "五陂下试剂剩余3天，建议本次携带更换" |
| 高危优先 | 计算站点优先级分 | "麻山优先级最高（紧急工单+健康度下降），建议排在周一/周二" |
| 路线折返 | 检测当天站点顺序 | "周三路线有折返，建议调整为星子→山口岩→袁河，省35km" |
| 覆盖遗漏 | 周巡检校验 | "本周未覆盖星子站，周巡检要求全覆盖" |

> 这些建议在**排程端**辅助运维排得更好，在**审批端**辅助管理者判断合理性（见第五节）。一套能力，两个场景。

### 3.3 出发前资源协调

**用车：**
```python
def check_vehicle_conflict(vehicle_id, period_start, period_end, user_id):
    """检查车辆在周期内是否被他人预约"""
    return SELECT date, user_name FROM vehicle_applications
           WHERE vehicle_id=? AND status IN ('approved','pending')
           AND user_id != ? AND date BETWEEN ? AND ?
```
- 支持 `vehicle_days` 不同天选不同车（解决冲突）
- 冲突时实时提示"赣A12345 周三已被张三预约"

**备件：**
```python
def suggest_parts(site_ids):

    suggestions = []
    for site_id in site_ids:
```

### 3.4 状态机（含变更）

```
draft → submitted → approved → modifying → change_submitted → approved(v+1) → archived
                  ↘ rejected（修改后重交）
```

- **变更场景：** 车辆故障、突发应急、天气、人员调配
- **变更通过后任务同步：** 全量重建+保护已完成（有提交记录的不动，未执行的按新计划重建）

## 四、审批决策支撑（管理者PC端）

> 核心：让管理者审批时不用翻历史记录，系统主动呈现"这个排法合不合理"。

### 4.1 站点情况卡（每个排入站点一张）

```
┌─────────────────────────────────────────┐
│  麻山站              健康度 91.5% ⚠       │
│  📍 距上一站(五陂下)：18km                │
│  🔴 未关工单：1条（氨氮超标，已9天，临近SLA）│
│  🟠 近期告警：2条（pH偏低×2）             │
│  🔧 上次巡检：7/15（7天前）正常           │
│  🧪 试剂：总磷剩余3天 ⚠                  │
│  📉 健康度趋势：96%→93%→91.5% ↓         │
│  💡 优先级：高（建议排前面）              │
└─────────────────────────────────────────┘
```

数据来源聚合（现有接口已具备，需聚合）：工单、告警、巡检记录、试剂库存、数据健康度。

### 4.2 风险预警（自动扫描计划）

**① 高危排序预警：**
```python
def check_priority_order(schedule):
    warnings = []
    for date, day_data in schedule.plan_data.items():
        for site_id in day_data["sites"]:
            score = compute_urgency_score(site_id)  # 工单等级+SLA+告警+健康度趋势
            if score >= HIGH and date_index(date) >= MID_WEEK:
                warnings.append(f"{site_name}优先级高但排在{date}，建议提前")
    return warnings
```

**② 路线折返预警：**
```python
def check_route_backtracking(schedule):
    warnings = []
    for date, day_data in schedule.plan_data.items():
        sites = day_data["sites"]
        coords = [get_site_coord(s) for s in sites]
        # 检测方向反转：A→B→C 中 C 比 B 更靠近 A
        for i in range(len(coords)-2):
            if distance(coords[i], coords[i+2]) < distance(coords[i], coords[i+1]):
                warnings.append(f"{date}路线折返，建议调整顺序，预估省{saved_km}km")
    return warnings
```

> 边界：只做简单折返检测+顺序建议，不做复杂TSP最优解。站点少（一天2-4个），算法当"提醒"而非"决策"，最终顺序人来定。

### 4.3 路线可视化

审批界面配小地图，切换日期显示当天站点连线，折返处标红，显示总里程和优化建议。

### 4.4 审批操作

```
[通过]  → 一键批准，触发自动流转
[退回]  → 填原因，退回修改
[调整]  → 管理员直接改（调换站点/日期），保存后重新校验通过
```

审批时看到：站点情况卡 + 风险预警清单 + 路线地图 + 资源明细 + 备注。

## 五、执行层与拍照/审核

### 5.1 任务生成（审批通过后）

```python
def generate_tasks(schedule_id):
    schedule = get_schedule(schedule_id)
    if schedule.tasks_generated: return
    for date, day_data in schedule.plan_data.items():
        for site_id in day_data["sites"]:
            # 按 schedule_type 匹配对应检查项模板
            template = get_template(site_id, schedule.schedule_type)
            create_insp_plan(schedule_id, date, site_id, template)
    mark_tasks_generated(schedule_id)
```

### 5.2 拍照清单（移动端，让运维清楚拍什么/拍多少/缺什么）

每个检查项预定义"应拍清单"：

```
┌─────────────────────────────────────┐
│  检查项：氨氮分析仪运行状态           │
│  需拍 3 张：                         │
│  ┌────┐ ┌────┐ ┌─ ─ ─┐              │
│  │ ✓  │ │ ✓  │ │ +   │ ← 缺：试剂余量│
│  │仪器│ │数据│ │试剂  │   (虚线框)   │
│  │面板│ │曲线│ │余量  │              │
│  └────┘ └────┘ └─ ─ ─┘              │
│  进度：2/3                          │
│  [异常项必拍，否则不能标记正常]        │
└─────────────────────────────────────┘
```

- 每检查项显示应拍内容（图标+文字）、已拍缩略图、未拍虚线框
- 顶部实时进度"本站需拍24张，已拍18张，缺6张"
- 提交前一目了然，不靠提交时报错

### 5.3 异常驱动审核（PC端为主，给双方减负）

**核心反转：默认全部通过，只标记异常，其余自动通过。**

```
┌────────────────────────────────────────────────────────┐
│  巡检照片审核 - 张三 7/22 麻山站（共48张）  [一键通过全部]│
├────────────────────────────────────────────────────────┤
│  按检查项分组，展开所有缩略图：                          │
│                                                        │
│  ▼ 氨氮分析仪（3/3张）                                 │
│  ┌────┐ ┌────┐ ┌────┐                                 │
│  │ ✓  │ │ ⚠  │ │ ✓  │  ← 点击=标记异常(红框)，再点取消  │
│  └────┘ └────┘ └────┘                                 │
│  ...                                                  │
│  已标记异常：1张 ｜ 未标记：47张（将自动通过）           │
├────────────────────────────────────────────────────────┤
│  [ 确认：通过47张，驳回1张 ]（驳回需填原因）            │
└────────────────────────────────────────────────────────┘
```

**交互逻辑：**
1. 进入审核页，所有照片默认"待通过"
2. 点击问题照片→加红框异常标记，再点取消
3. 底部按钮"通过N张，驳回M张"，一次确认完成
4. 驳回照片填原因（模糊/拍错/未拍到关键/其他）
5. 辅助：框选多张、键盘左右切换+空格标记、缩略图懒加载+点击放大

**效果：48张照片只有1张问题时，只需点2次（点问题照片+确认），而非48次"通过"。**

**可选增强（后期）：**
- AI预筛：模糊/过暗/过曝自动标"建议复核"
- 异常项照片优先排序（标记异常的检查项照片排最前）

## 七、双端职责

| 环节 | PC端（管理者） | 移动端（一线） |
|------|--------------|--------------|
| 排程 | 辅助（大屏排月/季/年计划） | **主**（排周计划+资源确认+提交） |
| 智能建议 | 审批时呈现 | 排程时实时提示 |
| 审批 | **主**（站点卡+风险预警+路线地图+通过/退回） | 移动审批（精简） |
| 资源协调 | 车辆台账/备件库存/审批 | 选车/选备件/确认 |
| 执行 | — | **主**（打卡+检查+拍照+工单+试剂+备件） |
| 照片审核 | **主**（异常驱动批量审核） | 移动审核（精简） |
| 变更 | 审批变更 | 发起变更 |

| 阶段 | 内容 | 估时 |

## 九、兼容性处理

| 场景 | 处理方式 |
|------|---------|
| 现有 weekly_inspection_plans 表 | 数据迁移到 inspection_schedules（schedule_type='weekly'），旧表保留只读 |
| 现有 insp_plans 执行层 | 不动，由调度层生成时关联 schedule_id |
| 旧模板无 schedule_type | 默认 'weekly'，逐步补充月/季/年模板 |
| 移动端旧版本 | insp_plans 格式不变，移动端无感知 |
| 路线优化算法 | 仅做折返检测，不做TSP，避免过度设计 |

## 十、关键设计取舍说明

1. **周计划并入巡检体系**：避免运维理解"周计划"和"巡检"是两个东西，管理者也不用在两个页面切换。统一模型，按频次分层。

2. **资源协调前置**：用车/备件在排程/接单后、出发前落实，符合"驻地统一、站点分散"的实际。

3. **智能建议双端复用**：同一套站点情况+风险预警+路线优化能力，排程端辅助排好、审批端辅助判断，不重复建设。

4. **审批决策支撑而非决策替代**：系统给提醒（哪里不合理、建议怎么调），最终拍板是人。路线优化只做折返检测，不做复杂路径规划。

5. **异常驱动审核**：把"逐张点通过"反转为"只标记异常其余自动通过"，因为绝大多数照片正常，给审核者减负。

6. **变更用全量重建+保护已完成**：当前规模（5-10人、周任务≤15条）下比增量同步简单稳妥，守住"有提交记录的不动"底线即可，后期规模扩大再升级。
